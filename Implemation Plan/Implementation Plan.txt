# [Goal Description]

사용자 로그인 시스템을 도입하고, 로그인한 사용자별로 이미지 생성 토큰(크레딧)을 관리하는 방안에 대한 수행 계획입니다. 프로젝트의 향후 확장성(Next.js 풀스택 전환 여부, 백엔드 구성 방안, 상용화 여부)을 고려하여 총 6가지 전략(로그인 3개, 토큰 관리 3개)을 제시합니다.

## User Review Required

> [!NOTE]
> 현재 프로젝트는 **Vite + React (순수 프론트엔드)** 구조입니다.
> 결제 연동이나 안전한 토큰 제어를 위해서는 API 키를 숨기고 DB와 통신할 수 있는 **백엔드(Backend) 서버** 또는 **Next.js/Remix 같은 풀스택 프레임워크** 도입이 필수적으로 반영되어야 합니다.

아래 옵션들을 검토하신 후 원하시는 방향(예: "Option A + Option 2" 조합)을 선택해 주시면, 해당 방향에 맞춰 실제 코드 구조 변경 및 패키지 도입 작업을 시작하겠습니다.

---

## 1. 로그인 시스템 도입 방안 (3가지 옵션)

### Option A. Supabase Auth (가장 추천하는 구조)
*   **설명:** 오픈소스 Firebase 대안인 Supabase를 사용하여 사용자 인증 및 데이터베이스를 동시에 구축합니다.
*   **장점:**
    *   인증(Auth)과 토큰을 저장할 DB(PostgreSQL)가 완벽하게 통합되어 있어, 가입 즉시 토큰 수를 제어하고 동기화하기 가장 쉽습니다.
    *   무료 정책이 넉넉하며, 소셜 로그인(구글, 애플 등) 연동이 클릭 몇 번으로 끝납니다.
*   **단점:** 초기 SQL 테이블 구조(Schema) 설정과 환경변수 설정 등 약간의 백엔드 이해도가 필요합니다.

### Option B. Clerk (가장 빠르고 미려한 UI)
*   **설명:** 최근 React 생태계에서 가장 각광받는 SaaS 인증 솔루션입니다.
*   **장점:**
    *   사전 제작된 아름다운 로그인 UI 모달(`<SignIn />`, `<SignUp />`)을 한 줄의 코드로 삽입할 수 있습니다.
    *   개발 경험(DX)이 압도적으로 뛰어나며, 소셜 로그인 셋업이 제일 쉽습니다.
*   **단점:** 외부 솔루션이므로, 사용자의 토큰 개수를 관리하려면 별도의 자체 DB(PlanetScale, MongoDB 등)를 구축하여 Clerk의 Webhook과 연동해야 하는 작업이 추가됩니다.

### Option C. Firebase Auth (가장 전통적이고 안정적인 방식)
*   **설명:** 오랫동안 검증된 구글의 Firebase Auth와 Firestore(NoSQL DB)를 결합합니다.
*   **장점:** 모바일 웹과 호환성이 뛰어나며 참고할 레퍼런스와 한글 문헌이 가장 많습니다.
*   **단점:** NoSQL 쿼리에 익숙하지 않으면 추후 결제 내역 저장이나 사용자 토큰 히스토리 추적 같은 관계형 데이터를 모델링할 때 다소 껄끄러울 수 있습니다.

## 2. 로그인 기반 토큰 관리 시스템 도입 방안 (3가지 옵션)

### Option 1. 백엔드 DB 직접 제어 (기본형)
*   **설명:** 로그인 시 DB(예: Supabase)의 `users` 테이블에 생성된 고유 ID(UUID)를 기반으로, `tokens` 수량을 저장한 열(Column)을 만들어 직접 관리하는 방식.
*   **로직:**
    1. 가입 시 웰컴 토큰 (예: 5개) 부여.
    2. 프론트엔드에서 GENERATE 버튼 클릭 -> 백엔드(또는 Edge Function)로 요청.
    3. 백엔드에서 토큰이 1개 이상 있는지 잔여량 검사.
    4. 토큰을 -1 차감하고 Stability AI API 호출.
    5. 생성된 결과물을 돌려줌.
*   **장점:** 가장 안전하고 정석적인 방법. 프론트에서 API KEY를 노출할 이유가 전혀 없어짐. 우회 불가능.
*   **단점:** 현재의 Vite(프론트만 있는 상태)를 버리고, Next.js나 별도의 Express API 서버를 구축해야 하는 대공사가 필요함.

### Option 2. Client-Side + Local DB 캐싱 (가벼운 B2B/데모 방식)
*   **설명:** 서버 없이 오직 프론트엔드 내 브라우저 로컬 저장소(localStorage/IndexedDB)와 Auth Provider의 커스텀 Claims를 엮어 사용하는 임시 방편.
*   **로직:**
    1. 로그인한 유저의 정보(Session) 내에 기본 제공 토큰 횟수를 기록해 둠.
    2. 프론트엔드 단에서 이미지를 만들 때마다 로컬 카운트를 올려 차감.
*   **장점:** 현재의 Vite 기반 프론트엔드를 그대로 활용 가능. 백엔드 서버나 복잡한 DB 설정이 불필요.
*   **단점:** 해커나 악의적인 유저가 브라우저 캐시를 조작하여 무한정 토큰을 쓸 위험성이 있음. 철저한 과금 시스템에는 부적합하며 알파 테스트용 데모에만 사용.

### Option 3. Stripe/결제 솔루션 연동 Token Webhook (상용화 패키지)
*   **설명:** 궁극적인 SaaS 과금 모델. 유저가 결제를 진행하면 외부 결제 모듈이 내 DB에 토큰을 자동으로 충전시켜 주는 방식.
*   **로직:**
    1. 토큰 소진 시 "토큰 구매" 모달 표시.
    2. Stripe Checkout으로 결제 진행.
    3. 성공 시 Stripe가 내 백엔드로 Webhook 신호를 보냄.
    4. 백엔드가 해당 유저의 DB 토큰을 +100 충전하고 알림을 보냄.
*   **장점:** 진정한 의미의 수익 창출형 B2C 프로덕트 구성을 완성할 수 있음. 자동화 시스템.
*   **단점:** Option 1의 완벽한 백엔드가 전제조건으로 깔려 있어야 하며, 국내 결제의 경우 포트원(Portone), 토스페이먼츠 연동을 위해 더 무거운 서버 비즈니스 로직 설계가 필요.
